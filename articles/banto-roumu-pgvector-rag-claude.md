## 背景：キーワード検索では「言い方の違い」に負ける

番頭は労務相談に答える個人開発のSaaSです。差別化の核は「自社の労務履歴を覚えていて、それを踏まえて答える」ことにあります。会社ごとに就業規則や過去の相談内容、過去に下した判断（決定）を記憶として溜め、次の相談ではその記憶を踏まえてClaudeに答えさせます。

最初期はキーワードと2-gramの部分一致で関連する記憶を引いていました。動くことは動くのですが、すぐに壁にぶつかりました。ユーザーが「産休明けの休みってどう扱えばいい？」と聞いても、記憶に「育児休業」という言葉でしか残っていないと拾えません。労務相談は言い回しのバリエーションが多く、同義語や言い換えに弱い検索は実運用でかなり効いてきます。

この記事では、この課題に対してpgvectorによる意味検索を後付けし、検索結果をClaudeのコンテキストに注入するところまでの実装判断を書きます。既存のキーワード検索を壊さずに追加する、という制約が設計のほとんどを決めています。

## 全体の流れ

1. 会社の記憶（相談要約・過去の判断・規程原文の抜粋）をテキスト化
2. `embedTexts` でOpenAIのtext-embedding-3-smallに投げてベクトル化
3. Supabaseのpgvectorに `vector(1536)` として保存
4. チャット時にユーザーの質問文も埋め込み、コサイン類似度で上位チャンクを検索
5. 検索結果を非PIIラベルに整形しつつ、原文はClaudeのsystemプロンプトに注入
6. 応答と一緒に「何を想起したか」をヘッダーで可視化

このうち3と4はSQL側の話なので割愛し、ここではlib配下の実装判断を中心に書きます。

## embedding.ts：SDKを足さずfetchで叩く

埋め込み層で最初に決めたのは「依存を増やさない」ことです。OpenAIの公式SDKを入れれば数行で書けますが、Vercelの関数サイズとコールドスタートに響きます。番頭はこの手のAPIラッパーをすべてfetchで自前実装する方針にしています。

```ts
const OPENAI_EMBEDDINGS_URL = 'https://api.openai.com/v1/embeddings'
const MAX_INPUT_CHARS = 8000
const EMBED_TIMEOUT_MS = 8000

export function embeddingEnabled(): boolean {
  return Boolean(process.env.OPENAI_API_KEY)
}
```

もう一つの判断が「グレースフルデグレード」です。番頭は個人開発のSaaSなので、埋め込みAPIの障害やAPIキー未設定でチャット機能全体を止めるわけにはいきません。`embeddingEnabled()` がfalseなら呼び出し側は最初からネットワークを叩かず、従来のキーワード検索にフォールバックします。API呼び出しが失敗・タイムアウトした場合もthrowせず、nullを返すだけにしています。

これによって「意味検索が使えないときは黙ってキーワード検索に戻る」という契約を型のレベルで保証しているわけではありませんが、運用としては一貫させています。呼び出し側でnullチェックを一箇所書けば済むので、RAG層全体の複雑さを抑えられました。

タイムアウトを8秒に設定しているのも、想起処理がユーザーとの対話の律速になることを避けるためです。埋め込みAPIが遅延しても、8秒待ってダメならキーワード検索に切り替えて応答自体は返す、という判断にしています。

次元数は `EMBEDDING_DIM = 1536` を定数として持ち、Supabase側のマイグレーション（`vector(1536)`）と厳密に一致させる必要があります。ここがずれると挿入・検索が両方失敗するので、コメントで警告を強めに書いています。将来Voyageなど別プロバイダに差し替える可能性を考えて、次元数の変更点をこのファイルとSQLの2箇所だけに閉じ込めているのも意図的な設計です。

pgvectorへの受け渡しはJS配列そのままではなく、テキストリテラルに変換して渡します。

```ts
export function toVectorLiteral(v: number[]): string {
  return `[${v.join(',')}]`
}
```

supabase-jsでinsertやRPCの引数にJS配列を渡すとJSONとしてシリアライズされてしまい、SQL側で `::vector` にキャストできません。そのため一度文字列化してから渡し、SQL側でキャストする形にしています。地味な話ですが、ここでハマると原因がわかりにくいので明記しておく価値があると思います。

## チャンク設計：規程原文は「まるごと」で持つ

チャンク分割の設計は、対象データの性質によって割り切りを変えています。

会話の要約（company_memories）や過去の判断（decision）は元々短い単位で記録されているので、そのまま1レコード=1チャンクとして埋め込みます。分割ロジックを別途持つ必要がありません。

一方、就業規則などの規程原文（company_documents）は事情が違います。

```sql
content text NOT NULL CHECK (char_length(content) <= 200000),
char_count int NOT NULL DEFAULT 0,
UNIQUE (company_id, title)
```

規程は「第◯条にこうある」と原文を引用して答える価値が大きいテキストです。ここを固定長で機械的にチャンク分割すると、条文の途中で切れて意味が壊れるリスクがあります。番頭では規程1本を最大100,000字（DB側では200,000字を天井に）許容し、チャット時には `loadRelevantRuleExcerpts` で関連しそうな抜粋を都度切り出す設計にしています。つまり「保存はまるごと、注入は抜粋」という二段構えです。これによりチャンク分割の粒度をあらかじめ決め打ちせず、質問内容に応じて動的に絞り込めるようにしています。

RLSも既存の `is_company_admin` / `is_company_member` を再利用し、新規のSECURITY DEFINER関数を作らない方針にしました。読み取りはメンバー全員、書き込みは管理者のみという権限モデルは他のcompany系テーブルと揃えており、テナント分離の考え方を1箇所に集約しています。

## 注入トークン量の制御：件数上限とラベル化

Claudeへの文脈注入で一番気を使ったのは、トークン量とプライバシーの両立です。

```ts
const MAX_PROFILE_ITEMS = 6
const MAX_RULE_ITEMS = 4
const MAX_DECISION_ITEMS = 5
```

`recall.ts` はチャットのsystemに実際に注入した内容をユーザーに可視化するための整形層です。ここでの判断は「氏名や相談本文などPIIになりうる情報は載せない」ことです。載せるのは規程名（title）、自社ルールのkey、判断のtopicラベルと決定日だけにしています。相談記憶（memory）は件数のみで、中身は出しません。

これは注入トークン量の抑制とも一致しています。Claudeのsystemプロンプトに大量の生データを詰め込むとコストとレイテンシが跳ねるので、件数上限を各カテゴリごとに設け（自社ルール6件・規程出典4件・判断5件）、それ以上は切り捨てるようにしています。可視化用のペイロードもレスポンスヘッダー（`X-Recalled-Memory`）で運ぶ軽量な設計にしているため、件数上限を厳格にしないとヘッダーサイズが膨らんでしまう事情もありました。

```ts
export function serializeRecalledMemory(r: RecalledMemory): string | null {
  const hasAnything =
    r.profileCount > 0 || r.memoryCount > 0 || r.decisionCount > 0 || r.ruleDocCount > 0
  if (!hasAnything) return null
```

想起した内容が0件ならnullを返してヘッダー自体を付けない、という細かい分岐も入れています。「何も思い出していないのに想起表示だけ出る」という違和感を防ぐための工夫です。

## チャットルートでの組み立て

`/api/company/chat` は会社IDの解決、プランに応じた利用回数のガード、Claudeへのストリーミング応答という一連の流れの中に、記憶注入を挟み込んでいます。

```ts
const MAX_MEMORIES = 10
```

`loadCompanyContext` で会社のプロファイル・記憶・判断を読み、`loadRelevantRuleExcerpts` で規程の関連抜粋を読み、両方を `buildCompanySystemPrompt` に渡してsystemプロンプトを組み立てる、という順序になっています。ここで埋め込みが有効かどうかの分岐（`embeddingEnabled()`）を先に見て、意味検索が使える場合と使えない場合でどちらの経路で記憶を集めたかを `buildRecalledMemory` の第三引数（semantic）に渡し、想起の出自をユーザーに見える形で区別できるようにしています。

高コストなsonnet呼び出しの前に日次利用上限のガードを入れているのも、個人開発ゆえのコスト管理です。DBマイグレーション未適用時はfail-open（許可する側）にしてあり、運用の安全弁として機能させています。

## まとめ

RAGを労務ドメインに実装する中で効いたのは、「壊さない」ための判断の積み重ねでした。埋め込みAPIが落ちてもキーワード検索に自動で戻す、次元数の不一致を起こさないようコメントで強く警告する、規程原文は分割せず保存して注入時だけ抜粋する、PIIを含みうる情報は可視化から意図的に外す。派手な工夫ではありませんが、個人開発でSaaSを回し続けるにはこの種の地味な防御線が積み上がって初めて、安心して機能追加ができるようになると感じています。

pgvectorはSupabaseに乗せるだけで導入コストが低く、既存のRLS設計とも相性がよいので、既存のPostgresベースのSaaSに意味検索を後付けする選択肢としてはかなり現実的だと思います。
