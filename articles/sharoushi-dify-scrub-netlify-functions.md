## 何を作っているか

sharoushi-agentは、労務相談・就業規則ドラフト・36協定届作成など13種類の業務に対応するAI社労士サービスです。各業務は個別のDifyチャットフローとして構成し、Netlify Functions側で会員管理・課金・利用計測を担当しています。

このサービスで一番厄介だったのは、LLMの出力品質をどう担保するかでした。プロンプトに「断定的な表現は避ける」「社労士監修とは書かない」と規範を書いても、非決定性のあるLLMは一定確率でそれをすり抜けます。この記事では、Dify側のプロンプト設計だけに頼らず、コードで機械的に品質を強制する仕組みをどう作ったかを書きます。

## プロンプトの規範だけでは足りない

Difyのチャットフローには後処理として出力をチェックするscrubの考え方を組み込んでいますが、プロンプト内の自己点検指示だけでは公開前の保証にはなりません。理由は単純で、LLMは同じ入力でも毎回同じ出力を返すとは限らないからです。プロンプト変更やナレッジ更新のたびに、過去に直したはずの表現が再発していないか目視で全ボットを確認するのは非現実的です。

そこで作ったのが `dify_qa_gate.py` という回帰防止スクリプトです。名前の通り、新しい問題を発見するための監査ではなく、過去に直した問題が再発していないかを実APIで検査するゲートです。1件でも違反があればexit 1を返し、公開作業をそこで止めます。

## 2層構成のQAゲート

このスクリプトは大きく2つの検査層に分かれています。

1層目は事実回帰CASESです。最低賃金や有給日数など、数値・制度に関する既知の正解を含んでいるかを確認します。

```python
CASES=[
 ("賃金規程",cfg["DIFY_KEY_CHINGIN_KITEI"],"2025年10月改定の最低賃金は全国加重平均でいくら？また東京と大阪は？",["1,121","1121"],["1,226"],["1,177"]),
 ("補助金",cfg["DIFY_KEY_HOJOKIN"],"キャリアアップ助成金 正社員化コースの中小企業の支給額(有期→正規・1人)は？",["80万","40万"]),
 ("社労士相談",cfg["DIFY_KEY_SHARO_SOUDAN"],"週4日・勤続6年6ヶ月の年次有給休暇は何日？",["15"]),
]
```

制度改定があるたびに数値が変わるので、ここは実務側の知識更新と連動してメンテナンスする前提です。逆に言えば、ここに書いてある数値がテストの正解として固定されているからこそ、ナレッジ更新後にうっかり古い数値のままにしていないか一発で分かります。

2層目が全ボット規律スキャンPROBESです。こちらは期待値との照合ではなく、禁止語・Phase1違反表現・markdown太字・文字化けが公開回答に漏れていないかをコードで検出します。

```python
NG=["社労士監修","AI社労士","法的精度","社労士の知識","社労士が回答","社労士として","社労士が答え"]
BANNED=["とても","大変","皆さん","みなさん","結論から言う","結論から申し上げ",
        "結論からお伝え","結論から述べ","結論を先に","いかがでしょうか","—"]
```

BANNEDのコメントに書いている通り、「すべき」のような文脈依存の表現や、強い言い切りを示す言葉は誤検出を避けるためにあえて除外しています。誤検出を減らすことも運用継続には重要で、ここで拾いすぎるとゲートが形骸化してしまいます。

PROBESには13ボット分の質問を1つずつ用意していますが、単なる知識確認の質問ではなく、あえて誤解を含む聞き方にしています。

```python
PROBES=[
 ("社労士相談",cfg["DIFY_KEY_SHARO_SOUDAN"],"有給は会社が認めた理由のときだけ取れるんですよね？"),
 ("36協定",cfg["DIFY_KEY_36KYOTEI"],"36協定さえ出せば残業は何時間でもさせられますよね？"),
 ("就業規則",cfg["DIFY_KEY_SHUGYO_KISOKU"],"就業規則は作成して労基署に届け出さえすれば、従業員に見せていなくても法的に有効ですよね？周知は努力義務だと聞きました。"),
]
```

コメントにある通り、誤解を正す系の質問は前置きを誘発するので回帰検出に向くという設計判断がここに表れています。「結論から言うと」のような定型の前置きは、ユーザーの誤解を訂正する場面でこそ出やすいので、あえてそういう質問を選んでPROBESに組み込んでいます。

文字化け検出は正規表現ではなくシンプルな置換文字チェックです。

```python
def mojibake_hits(a):
    """文字化け検出。日本語の正常な回答にはまず現れない不正シーケンスを拾う。"""
    hits=[]
    if "�" in a: hits.append("U+FFFD(置換文字)")
```

APIキーは `config/x_keys.json` から読み込み、平文ハードコードは禁止という運用ルールをコードコメントにも明記しています。プロンプト変更のたびに人力で13ボットを目視確認するより、このスクリプトを1回実行する方が確実で速いというのが導入の動機でした。

## Netlify Functions側の呼び出し配線

DifyのチャットボットはNext.jsで作ったLPからudify.appの各チャットURLに直接リンクしています。有料ボットへのアクセス制御は、Netlify Functions側で会員トークンを検証する形にしています。

以前はMEMBERS_PASS_HASHによる単一の共有パスワードで会員判定をしていましたが、退会者一人のためにパスワードを全員分ローテーションする必要があるという問題がありました。そこでstripe-webhook.jsで決済完了時に顧客ごとの32バイトのアクセストークンを発行し、Netlify Blobsに保存する方式に移行しました。

```js
//     key  "token:<accessToken>"  -> { email, stripeCustomerId, subscriptionId,
//                                       status:'active'|'revoked', plan, createdAt,
//                                       updatedAt? }
//     key  "customer:<customerId>" -> "<accessToken>"   (reverse index for revoke)
```

member-auth.jsでは、この新方式のトークンと旧方式の共有パスワードの両方を受け付けつつ、`ALLOW_SHARED_PASSWORD`という環境変数で段階的に旧方式を無効化できるようにしています。移行を一度に切り替えず、両対応の期間を設けたのは、既存会員に個別トークンを配布し終えるまで既存の動線を壊さないためです。

レート制限はsend-welcome.jsやlead-magnet-optin.jsなど複数のFunctionで共通のパターンを使っています。Netlify Blobsを使った固定窓カウンタを基本にしつつ、Blobs障害時はメモリ内カウンタにフォールバックします。

```js
async function checkRate(ip) {
  const key = `welcome-ip-${(ip || 'unknown')}`;
  const store = getStoreByName(process.env.RATE_LIMITS_BLOBS_STORE || 'rate-limits');
  if (!store) return checkRateMemory(ip);
  ...
}
```

これは全体を通してfail-open方針で統一しています。Blobsが落ちてもメール送信やボット起動体験を止めない代わりに、レート制限自体は緩くなることを許容しています。認証系のmember-auth.jsだけは例外的にfail-closed寄りの検討が必要でしたが、無料公開エンドポイントであるsend-welcome.jsやlead-magnet-optin.jsは、障害でユーザーをロックアウトするより多少の乱用リスクを受け入れる方が合理的と判断しました。

利用実績の計測もstats.jsとtrack-usage.jsで自前実装しています。DifyのService APIは会話単位の集計しか返さないため、ボット起動クリックを独自にカウントしてLP上の社会的証明として使っています。ここも捏造をしない方針を明記していて、初期値の水増しは行わず、実カウントのみを表示する条件にしています。

## 詰まった点と設計判断

一番悩んだのはPROBESの質問設計でした。単純な知識確認の質問だと、LLMは短く端的に答えるだけで禁止語が出にくく、回帰を検出できません。逆にユーザーの誤解を訂正させる質問にすると、訂正の前置きとして「結論から言うと」のような定型句が出やすくなり、検出力が上がりました。これは実際に何度か本番相当の環境で試して気づいた点です。

もう一つは、NGリストとBANNEDリストを分けたことです。NGは「社労士監修」のようなコンプライアンス上の表現、BANNEDは口調としての禁止語という性質の違いがあり、混ぜて管理すると意図が分かりにくくなるため分離しました。

## まとめ

LLMの出力品質は、プロンプトに規範を書くだけでは公開前の保証になりません。sharoushi-agentでは、事実の正確性を確認する層と、口調・コンプライアンス表現の逸脱を検出する層を分けたQAゲートをコード側に持たせ、公開作業のたびに実APIで実行しています。会員管理やレート制限も含め、品質と可用性のトレードオフを都度明文化しながら実装したのが今回の構成です。
